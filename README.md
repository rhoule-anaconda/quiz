# Security Team Quizzes

Multiplayer trivia for security team meetings, with a static-quiz mode for solo play. Deployed to Cloudflare Workers (with Durable Objects for game rooms) and gated behind Cloudflare Access (Anaconda SSO).

## Two modes

| Mode | URL | Use it for |
|---|---|---|
| **Multiplayer** | `/new` → host gets `/host/<code>` and `/play/<code>` | Live games during a Zoom — host advances questions, players answer on their phones, scores update on a shared dashboard |
| **Solo** | `/<slug>` (e.g. `/linux-sysadmin-security`) | Self-paced. Single page, no server state. The original mode. |

Both modes use the same `questions/*.json` files.

## Layout

```
questions/         # one .json per quiz — filename is the slug AND the title
src/
  template.html    # solo-mode quiz UI
  build.mjs        # questions/*.json → dist/*.html + src/worker/gen/quizzes.ts
  _headers         # security headers for static assets
  worker/
    index.ts       # Worker entrypoint: routes /new, /host, /play, /ws, /api/rooms
    game-room.ts   # GameRoom Durable Object (lobby → question → reveal → final)
    pages/         # HTML for /new, /host/<code>, /play/<code>
    gen/           # generated by build (gitignored)
dist/              # built static quizzes (served by the Worker's assets binding)
wrangler.jsonc     # Worker config — DO binding, static assets binding
```

## Authoring a new quiz

> **Solo-mode answers are visible in page source.** Multiplayer mode keeps answers server-side and only sends them at reveal time. So if cheating-resistance matters, use `/new` to start a hosted game; if it's casual, the static URL is fine.

1. Create `questions/<slug>.json`. Filename → URL slug → page title.
2. Each question: `cat`, `q`, `options[]`, `answer` (0-based), `why` (HTML allowed).
3. Optional `categories` block customizes label/icon/colors per category.

```json
{
  "categories": {
    "linux": { "label": "Linux", "icon": "🐧", "bg": "#1a4d22", "fg": "#8aff9a" }
  },
  "questions": [
    { "cat": "linux", "q": "...", "options": ["a", "b"], "answer": 1, "why": "..." }
  ]
}
```

## Build & develop locally

```sh
npm run build       # generate dist/ and src/worker/gen/quizzes.ts
npm run typecheck   # tsc --noEmit
npm run dev         # wrangler dev — full Worker + DO simulation locally
npm run preview     # open static dist/index.html in browser (no Worker)
```

`npm run dev` gives you the multiplayer flow at `http://localhost:8787` — including WebSockets and the DO state machine.

## Deploy

One-time:

```sh
wrangler login      # OAuth into the Anaconda Cloudflare account
```

Every deploy:

```sh
npm run deploy      # builds and ships
```

Cloudflare hands you `https://anaconda-security-quiz.<account>.workers.dev` (or your custom hostname).

## Access (auth)

Gated by Cloudflare Access on the Worker hostname. Anyone hitting it must complete Anaconda SSO before any HTML is served. App + IdP are configured in the Cloudflare Zero Trust dashboard.

Setup steps (one-time):
1. **Zero Trust → Access → Applications → Add a self-hosted application**
2. Application domain: the Worker hostname
3. Identity provider: existing Anaconda SSO IdP
4. Policy: allow `@anaconda.com` emails
5. Save

WebSocket connections to `/ws/*` go through the same Access gate — Cloudflare passes the JWT on the upgrade request, so multiplayer respects auth automatically.

## URL pattern

- `/` — entry page (type a quiz slug)
- `/<slug>` — solo-mode static quiz
- `/<token>` — full quiz listing (token in `.list-token`, gitignored)
- `/new` — host: pick a quiz, get URLs for the team
- `/host/<code>` — host dashboard (screen-share this during Zoom)
- `/play/<code>` — player join page

To rotate the listing token: `rm .list-token && npm run deploy`.

## Game lifecycle

A room's Durable Object goes through: `lobby → question → reveal → next/end → final`. After `final`, an alarm fires 2 hours later and wipes the room's storage. Rooms are addressed by code (e.g. `XAXA`) using deterministic `getByName(code)`. Codes are 4 chars from a 31-char alphabet, generated by the host on `POST /api/rooms`.

State is broadcast to all sockets after every change. Host sees full answers + per-option histogram + per-player completion; players see only what they should — never the answer key until reveal.
